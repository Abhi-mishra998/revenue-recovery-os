# Revora Data Schema

Single source of truth for every MongoDB collection. Keep in sync with
`backend/server.py` and `backend/services/*`. Models that train on this data
later will read against this contract — schema changes are versioned, not
silent.

## Conventions

- Every document has `id` (UUID string) — Mongo's `_id` is stripped from API
  responses.
- Every tenant-scoped document has `owner_id` (user UUID). Indexes always
  include `owner_id` first so cross-tenant reads are physically impossible
  to do efficiently by accident.
- Timestamps are ISO-8601 UTC strings unless noted. Server clock only.
- Money is `*_inr` floats in rupees (not paise — bigger ceiling, easier display).
- No PII goes to external services. The redactor at `services/ai/redact.py`
  tokenises emails / phones / PAN / GSTIN / Aadhaar before any LLM call.

## `users`

| Field           | Type    | Notes |
| --------------- | ------- | ----- |
| `id`            | string  | UUID |
| `email`         | string  | unique index, lowercased |
| `name`          | string  | display name |
| `auth_provider` | enum    | `email` \| `google` |
| `password_hash` | string? | bcrypt; null for google users |
| `token_version` | int     | incremented on logout to revoke outstanding JWTs |
| `created_at`    | iso8601 | |

## `proposals`
A sales proposal in flight.

| Field               | Type    | Notes |
| ------------------- | ------- | ----- |
| `id`                | string  | UUID |
| `owner_id`          | string  | tenant |
| `client_id`         | string  | FK clients |
| `title`             | string  | |
| `value_inr`         | float   | >0 |
| `sent_date`         | iso8601 | when proposal first sent |
| `last_contact_date` | iso8601 | drives auto-status |
| `stage`             | enum    | `sent` \| `negotiating` \| `won` \| `lost` |
| `outcome_at`        | iso8601? | timestamp the stage flipped to `won` or `lost`; null otherwise. **ML label** |
| `notes`             | string? | |
| `created_at`        | iso8601 | |

Derived (computed on read, never stored):
- `status` ∈ `active` (≤7d silent) \| `cold` (8-21d) \| `dead` (>21d)
- `days_silent`

Indexes: `(owner_id, 1)`.

ML notes: the (features, label) pair is `(value_inr, days_silent, industry, etc.) → (stage, outcome_at)`. Join with `followups` on `proposal_id` to attribute outcome to specific drafts.

## `followups`
**AI draft, persisted with full generation context.** Generated by
`POST /api/proposals/{id}/generate-followup`. One LLM call produces both the
WhatsApp text and the email, written as **two** rows (one per channel) sharing
the same `generation_id`.

| Field              | Type    | Notes |
| ------------------ | ------- | ----- |
| `id`               | string  | UUID |
| `owner_id`         | string  | tenant |
| `proposal_id`      | string  | FK proposals |
| `client_id`        | string  | FK clients (denormalised — saves a join later) |
| `generation_id`    | string  | UUID — pairs the WhatsApp + email rows from one call |
| `channel`          | enum    | `whatsapp` \| `email` |
| `draft_text`       | string  | the final, rehydrated, guardrail-passed text |
| `context`          | object  | full LLM input: sender, recipient, company, industry, title, value_inr, days_silent, redacted_keys |
| `prompt_ref`       | string  | e.g. `proposal_followup@v2` |
| `route_ref`        | string  | e.g. `simple:emergent_gemini/gemini-2.5-flash` |
| `confidence`       | float?  | self-reported by model, 0-1; nullable until v3 prompt |
| `latency_ms`       | int     | total wall time of the generation call |
| `created_at`       | iso8601 | |

Indexes: `(owner_id, proposal_id, created_at DESC)`.

ML notes: training pair = `(context + prompt_ref + draft_text) → (proposal.stage / outcome_at)`. Use `generation_id` to group multi-channel attempts.

## `events` (append-only)
**System-emitted signals.** Distinct from `activities` (which are
user-curated) and from `audit_log` (which is signed and tamper-evident). This
collection is for product analytics and ML feature engineering.

| Field         | Type    | Notes |
| ------------- | ------- | ----- |
| `id`          | string  | UUID |
| `owner_id`    | string  | tenant |
| `event_type`  | string  | dot-namespaced — see catalogue below |
| `entity_type` | string  | `proposal` \| `invoice` \| `client` \| `followup` |
| `entity_id`   | string  | |
| `prior_value` | any?    | for transitions (e.g. previous stage) |
| `new_value`   | any?    | for transitions (e.g. new stage) |
| `metadata`    | object  | event-specific extras |
| `source`      | enum    | `user` \| `system` |
| `created_at`  | iso8601 | |

**Event-type catalogue (extend as needed):**

| Event type                    | Fired by                                     | metadata fields |
| ----------------------------- | -------------------------------------------- | ---------------- |
| `proposal.created`            | POST /proposals                              | `value_inr`, `stage` |
| `proposal.stage_changed`      | PATCH /proposals with `stage` change         | (prior/new_value carry old/new stage) |
| `proposal.won`                | stage → `won`                                | `days_to_close`, `value_inr` |
| `proposal.lost`               | stage → `lost`                               | `days_to_close`, `value_inr` |
| `invoice.created`             | POST /invoices                               | `amount_inr`, `due_date` |
| `invoice.payment_received`    | PATCH /invoices with `paid_date` set         | `amount_inr`, `days_overdue_at_payment` |
| `followup.generated`          | POST /proposals/{id}/generate-followup       | `generation_id`, `prompt_ref`, `route_ref`, `latency_ms` |

Indexes: `(owner_id, entity_id, created_at DESC)`, `(owner_id, event_type, created_at DESC)`.

Append-only by convention: no PATCH/DELETE endpoint. If you ever need to retract, append a corrective event instead.

## `client_memory`
**Per-client derived features.** Recomputed (not patched) from current state
on every activity / event. Keeps the "what we know about this client" view
queryable for both the UI and future personalisation models.

| Field                  | Type    | Notes |
| ---------------------- | ------- | ----- |
| `id`                   | string  | UUID |
| `owner_id`             | string  | tenant |
| `client_id`            | string  | FK clients, **unique per (owner_id, client_id)** |
| `channel_preference`   | enum?   | `whatsapp` \| `email` \| `call` \| null — channel with the most inbound replies |
| `channel_counts`       | object  | `{whatsapp: int, email: int, call: int, meeting: int, note: int}` |
| `typical_response_days`| float?  | median days from outbound activity → next inbound activity |
| `response_rate`        | float?  | fraction of outbound activities followed by an inbound within 14 days |
| `last_outcomes`        | array   | up to last 10 entries — `[{type: "proposal.won"|"proposal.lost"|"invoice.paid", id, at}, …]` |
| `recompute_count`      | int     | bumped each rebuild — useful for debugging stale views |
| `updated_at`           | iso8601 | |

Indexes: `(owner_id, client_id)` unique.

Recompute is idempotent — running it twice on identical inputs produces the same document. Cheap to rebuild for one client (single-digit ms on a populated DB).

## `audit_log` (already shipped)
Signed, hash-chained, tamper-evident. See `services/audit.py`. Distinct from
`events`: audit_log is for security/forensics; events is for product analytics.

## Versioning

The shapes above are **v2** of the schema (the legacy v1 rename migration lives
in `migrate_legacy_fields`). Future breaking changes get a new migration
function and a bump to v3 — additive fields don't require a version bump.
